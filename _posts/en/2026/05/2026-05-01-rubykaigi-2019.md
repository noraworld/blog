#### The Year of Concurrency
The first session was Matz’s keynote. He talked about Ruby’s concurrency, types, MJIT, multicore, pattern matching, and the future of Ruby.

##### Ruby's Pros and Cons
Ruby is known for being productive, flexible, and fun to write, but it also has drawbacks like performance issues compared to other languages and not effectively utilizing multicore, as Matz explained.

The story of Twitter moving from Ruby (Rails) to Scala for performance reasons is well-known, but Matz pointed out that Ruby improving with each version, though it still has possible improvements to make. It's used in big companies like GitHub, Airbnb, and Cookpad, indicating its value.

Regarding performance, no language is significantly faster than others, said Matz. However, Ruby has traditionally been labeled as slow, he noted.

##### Testing
When the topic of testing came up, Matz humorously admitted, “I actually dislike testing,” which got some laughs from the audience. He mentioned that tests are not DRY and he doesn’t have an alternative to ensure code quality, so he grudgingly writes tests.

##### Type Definition Format
Then he talked about types. While other scripting languages have introduced type approaches, like PHP with type hinting, Python with type annotation, and JS with TypeScript, Matz is against embedding type declarations in Ruby. He dislikes type annotations because they aren't DRY and prefers to offer a way to define types in separate files (using `.rbi` extension) to gain the benefits of types.

He mentioned libraries like Sorbet and Steep for type definitions and checks.

##### Concurrency
Regarding concurrency, he described how other languages approach it and what Ruby plans to do. Languages like Erlang and Elixir were built with concurrency in mind, but Ruby cannot follow the same path because it cannot make all objects immutable now.

He also noted that Node.js is single-threaded but can handle many processes due to factors like the fast V8 engine and non-blocking I/O.

Among the approaches Ruby might take are Guilds (still an uncertain name, also referred to as Isolates) and Autofiber (also not finalized, sometimes called Whatever), although I don't fully grasp these concepts due to my technical limitations. They're still seeking a catchy name, and Guilds are unpopular in the gaming industry, Matz humorously added.

##### `?` String
I was surprised to learn about the `?a` syntax, where it returns a `"a"` string. It’s an Emacs inheritance, and Matz admitted that it wasn’t the best decision to include it, though this spec will remain in Ruby 3.

##### Keyword Arguments
Matz mentioned revising the keyword argument specification as its current implementation can be confusing and isn’t well-suited to typing. Although the backward compatibility might be broken, it’s a necessary step, he said.

##### Numbered Block Parameter
Ruby's new syntax allows implicit referencing of block arguments using `@1`, `@2`, etc. It’s already in the trunk and available in `2.7.0-dev`.

```ruby
[1, 2, 3].map do |i,|
  i * 2
end
# => [2, 4, 6]
```

This can be written as:

```ruby
[1, 2, 3].map do
  @1 * 2
end
# => [2, 4, 6]
```

I find it useful, though it’s controversial because some argue it's hard to read or looks like instance variables.

##### Pattern Matching
Freshly merged into the trunk, pattern matching uses `case` ... `in` syntax to simplify code.

##### Conclusion
After trying features like threads and keyword arguments, Matz acknowledged some regrets. Though hindsight is easy, some decisions are bound to be less favorable around the edges. Once features are implemented, they’re tough to change, and even if they seem unused, someone probably uses them, generating laughter from the audience.

A session with Cookpad's full-time committers, Sasada-san and Endo-san, among others, providing an update on Ruby 3's current developments.

They highlighted improvements in static analysis, developments related to the type definition format, the aim for being three times faster than 2.0, and integrating Bundler with RubyGems.

##### Watch Out for Account Takeovers on rubygems.org!
There have been numerous account takeovers on rubygems.org, with instances of malicious code insertion, as happened with Twitter's Bootstrap gem. They advised using two-factor authentication, not reusing passwords, and setting strong passwords to avoid this.

#### How to use OpenAPI3 for API developer
I realized I hadn’t mentioned it earlier, but RubyKaigi typically hosts four sessions simultaneously, and attendees choose which one they’re interested in. Matz’s keynote and the sessions by Sasada-san and Endo-san were single stand-alone events. You can see [RubyKaigi 2019's Schedule](https://rubykaigi.org/2019/schedule) for more details.

The sessions can either be in Japanese or English. Since I'm not proficient in English, I preferred Japanese sessions to ensure I wouldn’t miss the content.

In this session, led by [@ota42y](https://twitter.com/ota42y), I learned about OpenAPI. While I had heard the name, I wasn’t familiar with its details, so I attended the session out of curiosity.

Until attending, I mistakenly believed that OpenAPI was akin to REST, a new API specification, but it’s actually a format for documenting APIs. It enables machine-readable API definitions and prevents deviations between definitions and implementations.

##### Structure of OpenAPI 3
OpenAPI 3’s structure includes `info`, `paths`, `servers`, `components`, `security`, `tags`, and `externalDocs`. The `security` section allows for defining authentication rules like JWT, BASIC auth, API keys, or OAuth 2, but it’s just a definition, so specifying OAuth 2 doesn't automatically enable it, they clarified.

##### Using OpenAPI
OpenAPI describes API rules, ensuring that human errors—mismatches between defined functionality and actual behavior—are mitigated. One tool achieving this is the [Committee](https://github.com/interagent/committee) Gem, which checks for these mismatches and prevents invalid requests from reaching Rack applications.

Committee also enables response validation, notifying you of any discrepancies between expected outputs and actual ones. There’s an option to log these discrepancies without causing build terminations.

##### Differences between OpenAPI 2 and OpenAPI 3
Several differences exist between OpenAPI 2 and OpenAPI 3, though new elements like `security` seem to be the main additions.

##### Endpoint Validation with OpenAPI
A “Patricia tree” was introduced, demonstrating how paths are validated. For instance, `/users/meals/42/lunch` can be traced through `root -> meals -> {id} -> lunch`. If it doesn't reach the final segment, it’s determined as undefined and will be blocked by validation.

##### OpenAPI Tools
They introduced OpenAPI Generator and Swagger UI, which supports OpenAPI 3.

#### A Type-level Ruby Interpreter for Testing and Understanding
A session by Cookpad's Endo-san regarding types, continuing from Matz’s keynote. He delved into type profiler and type checking implementation.

##### Type Profiler
```ruby
def foo(n)
  n.to_s
end

foo(42)
```

When normally interpreted, the `foo` function is understood to take `42` and return `"42"`, but a type profiler sees it as taking `Integer` and returning `String`.

The type profiler might look like:

```
Object#foo :: (Integer) -> String
```

Above defines a method `foo` that receives `Integer` and returns `String`.

I might have incorrect notations as I couldn’t note them down precisely, so please just take it as the general gist of type profiling.

##### Is Type Checking Executed Every Time?
When methods are called, one might wonder if type-checking occurs every time. It's said to be skipped when unnecessary.

```ruby
def foo(n)
  n.to_s
end

foo(42)
foo(43)
foo(42.0)
```

For `foo(42)`, it’s the method’s first call, so type checking happens. In `foo(43)`, it already knows from `foo(42)` that it handles `Integer` and returns `String`, so it’s skipped. But `foo(42.0)` requires checking since it's `Float`, unexplored till then.

##### Recursive Calls
A Fibonacci sequence was used to illustrate.

```ruby
def fib(n)
  if n > 1
    fib(n - 1) + fib(n - 2)
  else
    n
  end
end
```

For this Fibonacci calculating method, type checking is initially pending inside the `if`, as the returned type is unknown. Seeing the `else`, it determines the method returns `Integer`. It results in a type profiler like:

```
Object#fib :: (Integer) -> Integer
```

##### Challenges with Type Profiler
Despite being beneficial, type profilers have implementation challenges. I couldn’t catch every mention, but unresolved implementations and state explosions were factors explained. When pending repeat, the states to consider multiply, causing delays.

Type profiler allows Ruby developers to receive type benefits without writing explicitly typed code. For those who wish to handle types specifically, they’re advised to use Sorbet or Steep, mentioned earlier.

#### RMagick, migrate to ImageMagick 7
A session by [@watson1978](https://twitter.com/watson1978) on improving RMagick.

RMagick lets you manipulate images through the ImageMagick library using Ruby, which I have used, so it’s somewhat familiar.

After a two-year hiatus post version 2.16.0, improvements by Watson-san led to the release of version 3.1.0. The session mainly detailed what enhancements were made since 2.16.0.

##### Setup Issues
Having personal experience, I know that RMagick setup can be tricky. The session discussed the setups causing issues and their resolutions.

RMagick, relying on ImageMagick 6, installs through Homebrew on macOS under special paths. Previously, RMagick couldn’t recognize these paths, leading to errors, but adjustments corrected this to recognize Homebrew’s paths correctly.

Windows setups also faced issues, primarily with MingW and MSWin, which have been fixed.

##### Memory Leak Issues
RMagick was infamous for causing memory leaks, and Watson-san reportedly resolved 56 instances. These 56 leaks fell into four patterns, each explained for their reasons.

Case 1 involved exceptions occurring between memory allocation and release, preventing proper memory freeing. Case 2 was due to having allocations without freeing, a straightforward error. Case 3 wasn’t noted due to unclear hearing. Case 4 stemmed from leaks on the ImageMagick 6 side, which were resolved by filing issues with ImageMagick.

##### How Memory Leaks Were Identified
Watson-san utilized Xcode's profiler tool, a real-time memory usage observation capability.

By observing memory-consumptive areas within code, he could read the suspect code to resolve the issues.

However, the tool’s limitation is its dependency on existing tests, pointed out Watson-san. Unfortunately, RMagick has too few tests, though they intend to incrementally increase them to surface more leaks.

##### Fixed SEGVs
Besides memory leaks, Watson-san fixed 5 SEGVs, out of which one remains workaround-ed.

Specifics eluded my understanding, but `ruby_stack_length()` indicating 47GB (this number may be wrong 🙇‍♂️) purportedly caused glyphs, as attempt to execute GC for it led to crashes.

Uncertainty exists whether the problem lay in Ruby, RMagick, or ImageMagick, hence the workaround remains.

##### Future Support
While RMagick currently relies on ImageMagick 6, they aspire to support ImageMagick 7. Three potential plans are outlined:

- Plan 1: Support ImageMagick 6 with RMagick 3.x and ImageMagick 7 with RMagick 4.x
- Plan 2: Use conditional logic (`if`) to differentiate between ImageMagicks 6 and 7
- Plan 3: Separate them using Bundler

Though Plan 2 appears convoluted — which Watson-san is reluctant to undertake due to added complexity — the community seems supportive. The plan for eventual support of ImageMagick 7 remains undecided.

Since RMagick is a Gem I use, I hope it becomes increasingly user-friendly.

#### Pattern matching - New feature in Ruby 2.7
A session by [@k_tsj](https://twitter.com/k_tsj) about the newly introduced pattern matching in Ruby, which Matz also mentioned. Considering how trendy a feature pattern matching is, I was intrigued to learn about it.

```ruby
case [0, [1, 2, 3]]
in [a, [b, *c]]
  p a # => 0
  p b # => 1
  p c # => [2, 3]
end
```

With the code above, pattern matching structures simplify inherent complexities. It currently uses the `case` ... `in` syntax, but this might change, as it's an experimental feature. Simplifying code is immensely rewarding.

```ruby
case [0, 1]
in [a, b] unless a == b
  :reachable
end
```

The example above shows conditions paired with pattern matching (`unless a == b`), known as "guards" in other languages.

##### Types of Patterns
Several pattern kinds were introduced:

- Value Pattern
- Variable Pattern
- Alternative Pattern
- As Pattern
- Array Pattern
- Hash Pattern

The value pattern matches specific values, as demonstrated below:

```ruby
case val
in 0
  p 'val is 0'
in 1
  p 'val is 1'
in 2
  p 'val is 2'
end
```

The variable pattern, as the name suggests, matches variables:

```ruby
case 1
in a
  p a # => 1
end
```

In instances where a matching variable already exists within the scope, it’s overlooked:

```ruby
a = 0

case 1
in a
  p a # => 1
end
```

However, prefixing the variable with a caret (`^`) prompts recognition:

```ruby
a = 0

case 1
in ^a
  :unreachable
end
# => NoMatchingPatternError
```

The above conveys `in ^a` becomes equivalent to `in 0`, thus it doesn’t match, resulting in `NoMatchingPatternError`.

An alternative pattern using `|` covers matching with either value:

```ruby
case 0
in 0 | 1 | 2
  :reachable
end
```

The as pattern (using `=>`) facilitates broader matching, akin to checking types, alongside specific values. Unless I misheard, comparison employs `===` similar to `case` ... `when`.

```ruby
case 0
in Integer => a
  p a # => 0
end
```

Array pattern and hash pattern engage Arrays and Hashes for pattern matching. For example, hash patterns can be applied as follows:

```ruby
case { a: 0, b: 1 }
in { a:, b: }
  p a # => 0
  p b # => 1
end
```

Though emphasizing array and hash patterns, I lack the confidence to explain thoroughly here. I recommend other articles or the soon-to-be-released videos for detailed insights.

##### Summary
During the pattern match introduction, the emphasis was on compatibility. A fresh reserve word, initially proposed as `match`, was dismissed to avoid breaking programs utilizing `match` as a variable, had it been reserved. Instead, `case` ... `in` mirrors the `case` ... `when` style.

Pattern matching originates from functional programming; thus, copying directly might lack the Ruby style, they noted. Leveraging Ruby's robust Array and Hash support harmonized implementation with Ruby conventions.
