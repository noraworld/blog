#### The Year of Concurrency
The first one was Matz's keynote. I listened to things about Ruby’s concurrency, types, MJIT, multicore, pattern matching, and the future of Ruby.

##### Ruby’s Pros and Cons
Ruby is productive, flexible, and fun to write. On the downside, it has performance issues (execution speed) and isn’t taking full advantage of multicore, especially when compared to other languages.

It’s well-known that Twitter shifted from Ruby (Rails) to Scala due to performance reasons, but just because Twitter left Ruby doesn’t mean Ruby is bad. Ruby has improved its performance with each version over time. It’s used by famous companies like GitHub, Airbnb, and Cookpad, showing it can still get better.

Matz mentioned that, although every language has its drawbacks, Ruby is often unfairly labeled slow.

##### Tests
When the topic shifted to tests, Matz humorously mentioned, "I actually don’t like tests," which drew laughs. He explained tests aren’t DRY (It's not DRY), but since no alternative method exists for quality assurance, he reluctantly continues using them.

##### Type Definition Format
The conversation then moved to types. While languages like PHP, Python, and JS have adopted approaches like type hinting, type annotations, and TypeScript to handle types, Matz is against having type declarations in Ruby. He dislikes type annotations because they're not DRY. He’s compromising with tests, but doesn’t want to compromise with types. Instead of embedding types directly in Ruby files, they're proposing separate files (with a `.rbi` extension) to define types, allowing Ruby to benefit from typing.

Sorbet and Steep were introduced as libraries for type definition and checking.

##### Concurrency
We then explored concurrency, comparing other languages and Ruby’s direction. Erlang and Elixir, for instance, have inherent concurrency. Ruby can’t mimic them because making every object immutable now isn’t feasible.

They discussed how Node.js is single-threaded yet effectively handles numerous tasks due to its fast V8 engine and non-blocking I/O strategies.

Go's goroutine was also mentioned.

Among these strategies in other languages, Ruby is introducing Guild (name not finalized, also referred to as Isolate) and Autofiber (possibly just a placeholder name referred to as Whatever). However, these concepts were challenging for me to grasp fully. They are still gathering ideas for names, and the gaming industry seems opposed to 'Guild' as a name!

##### `?` String
At some point, I was surprised to learn about the `?a` syntax that results in `"a"`. Derived from Emacs, Matz humorously regretted its inclusion. Nevertheless, this will remain even with Ruby 3.

##### Keyword Arguments
There's a plan to change the keyword argument specification because the current implementation can cause confusion and isn't well-aligned with typing. This change may lead to incompatibilities, though it's done reluctantly.

##### Numbered Block Parameters
This new feature, already part of the trunk and available in `2.7.0-dev`, simplifies how block parameters are handled. 

```ruby
[1, 2, 3].map do |i,|
  i * 2
end
# => [2, 4, 6]
```

can be written as

```ruby
[1, 2, 3].map do
  @1 * 2
end
# => [2, 4, 6]
```

Although I find it convenient, opinions are mixed; some say it’s hard to read, and the `@` symbol is perhaps misleading. The precise syntax is still being considered.

##### Pattern Matching
Just merged into the trunk before RubyKaigi, this new feature uses `case`...`in` for pattern matching.

##### Final Thoughts
Matz expressed regrets about some things like threads and keyword arguments he wishes he hadn't added. These situations were described as moments where foresight was lacking and once a feature gets in, removing it isn’t simple, even if very few people use it. This prompted laughter from the audience. Essentially, once a feature is in, it’s challenging to remove or change.

#### How to use OpenAPI3 for API developers
I forgot to mention that at RubyKaigi, four sessions are typically held at the same time, allowing you to choose the one you're interested in. Matz's keynote, and the session by Sasada-san, Endo-san, and others were the only one at that time, so everyone gathered at the same place. The schedule for [RubyKaigi 2019](https://rubykaigi.org/2019/schedule) shows it in more detail.

Sessions were sometimes in English or Japanese, but since I don't understand English well, I mostly chose Japanese sessions to avoid missing out on the content.

In this session by [@ota42y](https://twitter.com/ota42y), I learned about OpenAPI. I had heard of OpenAPI before but never really explored it, so picked this session out of curiosity.

Before the session, I mistakenly thought that OpenAPI was a new API specification similar to REST, but it turned out to be a format for describing APIs. It helps ensure the definition and implementation don’t diverge by allowing machine-readable API definitions.

##### OpenAPI 3 Structure
OpenAPI 3 includes sections like `info`, `paths`, `servers`, `components`, `security`, `tags`, and `externalDocs`. For instance, `security` allows you to describe authentication rules like JWT, BASIC authentication, API keys, or OAuth 2. It’s just a definition, meaning writing OAuth 2 here doesn’t automatically enable it.

##### Using OpenAPI
OpenAPI describes API rules but doesn’t itself implement anything. It helps define and ensure what parameters an API accepts and what it returns, thus ensuring the API’s real implementation matches its definition.

Tools like the [Committee](https://github.com/interagent/committee) gem can be used to validate APIs. It sits between client apps and Rack apps, ensuring requests and responses adhere to the API’s definition. If required parameters are missing, Committee filters them out before they reach the Rack application. Similarly, response validation helps catch mistakes by logging them without affecting production.

##### Differences Between OpenAPI 2 and OpenAPI 3
Several changes exist between OpenAPI 2 and 3, with new features like the `security` section being a key change.

##### OpenAPI Endpoint Validation
The PATRICIA tree was introduced. Imagine a tree structure:

```
    root
      |
    users
      |
    meals
   /    \
  me   {id}
  /      \
lunch  lunch
```

An endpoint like `/users/meals/42/lunch` travels `root -> meals ->{id} -> lunch`. If it doesn’t reach the leaf, it’s rejected during validation, indicating it doesn't match any defined route.

##### OpenAPI Tools
Tools like OpenAPI Generator and Swagger UI were also mentioned, with Swagger UI supporting OpenAPI 3.

#### A Type-level Ruby Interpreter for Testing and Understanding
This session by Cookpad’s Endo-san elaborated on types, following up on Matz's keynote. It delved into type profiling and checking implementations in detail.

##### Type Profiler
```ruby
def foo(n)
  n.to_s
end

foo(42)
```

A typical interpreter reads the above as `foo` takes `42` and returns `"42"`, while a type profiler interprets it as taking an `Integer` and returning a `String`.

A type profile appears like this:

```
Object#foo :: (Integer) -> String
```

indicating that the `foo` method accepts an `Integer` and returns a `String`.

While I didn’t capture every detail verbatim, this illustration conveys the type profiler's essence.

##### Frequency of Type Checking
Not every method invocation undergoes type checking, which skips where unnecessary.

```ruby
def foo(n)
  n.to_s
end

foo(42)
foo(43)
foo(42.0)
```

Using `foo(42)`, the initial `foo` call prompts a type check. Subsequent calls like `foo(43)` have the type check skipped, as `foo(42)` established receiving an `Integer` and returning a `String`. However, `foo(42.0)` invokes a type check since it’s an unexplored `Float`.

##### Recursive Calls
The Fibonacci sequence was the example.

```ruby
def fib(n)
  if n > 1
    fib(n - 1) + fib(n - 2)
  else
    n
  end
end
```

In this recursive Fibonacci, `fib` pending results in `if` due to unresolved return type expectations. Come `else`, revealing `Integer` returns, simplifies constructing the type profile:

```
Object#fib :: (Integer) -> Integer
```

##### Type Profiler Challenges
The type profiler faces implementation issues, such as being unfinished or state explosion (pending repeats doubling states), delaying analysis.

While profilers offer benefit without demanding manual type description, Sorbet or Steep can be opted for where explicit type handling is desired.

#### RMagick, migrate to ImageMagick 7
This session by [@watson1978](https://twitter.com/watson1978) explained enhancements made to RMagick.

RMagick is a gem enabling Ruby access to the image processing library ImageMagick. I was familiar due to previous use.

RMagick 2.16.0 saw development stagnation for nearly two years. With Watson-san's contributions, it improved significantly, releasing version 3.1.0.

The session primarily highlighted improvements since 2.16.0.

##### Setup Issues
RMagick can present installation hurdles. The problem along with solutions was described here.

Dependent on ImageMagick 6, installing on macOS via Homebrew results in installation to a special path. Old RMagick couldn’t detect this, leading to errors, which is now resolved.

Similarly, setup issues on Windows were addressed, resolving MingW and MSWin-related problems.

##### Memory Leak Issues
RMagick was known for memory leaks, 56 in total resolved by Watson-san. These fit 4 main patterns explored in detail.

Case 1: Memory allocation - exception occurs, preventing proper release, causing memory leaks.
Case 2: Allocates but forgets freeing the memory.
Case 3: Didn’t get this one accurately.
Case 4: ImageMagick 6 causes leaks. This was brought to ImageMagick who then fixed it. An example: memory is opened twice but released only once, causing leaks.

##### How Watson-san Identifies Memory Leaks
Watson-san reportedly uses Xcode’s profiler tool, which monitors real-time memory usage. Observing suspicious spike areas, the code is reviewed and solved.

However, as useful as it is, profiler tool only covers the tested areas. RMagick lacks ample tests, which they plan to increase, potentially uncovering new memory leaks.

##### SEGV Fixes
Besides memory leaks, SEGVs were also resolved. Of the five, four were resolved, with one remaining a workaround. 

Details proved challenging, but an instance was `ruby_stack_length()` pointing to a purported 47GB - running out of memory during GC led to crashes. Hopefully, I got this right.

The specific roots were unclear - whether it was Ruby-related, RMagick, or ImageMagick, hence the workaround.

##### Future Support
RMagick is dependent on ImageMagick 6, but plans to support ImageMagick 7. Three future paths were outlined:

- Plan 1: RMagick 3.x supports ImageMagick 6 while RMagick 4.x supports ImageMagick 7
- Plan 2: Use `if` statements to conditionally choose ImageMagick 6 or 7 based on environment
- Plan 3: Divide using Bundler

Plan 1 might lead to repeating failed installations with RMagick 4.x, causing reluctance. Plan 2, despite increased conditional complexity, seemed favored by colleagues though Watson-san wanted to avoid it.

A decision on how to proceed has yet to be made, but as an RMagick user, I’d welcome any improvements making it easier to use!

#### Pattern Matching - New Feature in Ruby 2.7
This presentation by [@k_tsj](https://twitter.com/k_tsj) covered the pattern matching mentioned also in Matz's keynote. It's a hot Ruby feature, so drew my attention.

```ruby
case [0, [1, 2, 3]]
in [a, [b, *c]]
  p a # => 0
  p b # => 1
  p c # => [2, 3]
end
```

The pattern matching shown above uses `case`...`in` syntax. An experimental feature in the trunk, syntax changes remain possible. It simplifies code considerably.

```ruby
case [0, 1]
in [a, b] unless a == b
  :reachable
end
```

Pattern matching introduces conditions (like `unless a == b`) that are comparable to "guards" in other languages.

##### Types of Patterns
Several pattern types were introduced:

- value pattern
- variable pattern
- alternative pattern
- as pattern
- array pattern
- hash pattern

A value pattern matches values.

```ruby
case val
in 0
  p 'val is 0'
in 1
  p 'val is 1'
in 2
  p 'val is 2'
end
```

A variable pattern matches variables.

```ruby
case 1
in a
  p a # => 1
end
```

The variable pattern rejects previously scoped variables, defaulting to:

```ruby
a = 0

case 1
in a
  p a # => 1
end
```

Prefacing with a caret (`^`) allows reference:

```ruby
a = 0

case 1
in ^a
  :unreachable
end
# => NoMatchingPatternError
```

The `in ^a` converts to `in 0`, preventing a match, causing a `NoMatchingPatternError`.

An alternative pattern uses `|` for conditional pattern matching.

```ruby
case 0
in 0 | 1 | 2
  :reachable
end
```

As patterns use `=>` for broader-type matching. The mechanism resembles `===` comparisons under `case`...`when`.

```ruby
case 0
in Integer => a
  p a # => 0
end
```

Array and hash patterns match using these respective data structures. An example is shown with hash patterns:

```ruby
case { a: 0, b: 1 }
in { a:, b: }
  p a # => 0
  p b # => 1
end
```

The significant focus was on array and hash patterns, though my explanation may fall short; I'd recommend further reading or checking videos to become available later.

##### Summary
Compatibility was prioritized in pattern matching implementation. A new keyword `match` was considered but ultimately dismissed to avoid breaking existing uses. Instead, they sought an alternative similar in form to `case`...`when`, settling on `case`...`in`.

As pattern matching stems from functional programming, a direct borrowing from those languages wouldn't fit Ruby well. The implemented design leverages Ruby's strong Array and Hash support, enhancing its integration.
